Slide 02 — Overview

AWS Services Used

Six AWS and third-party services working together to deliver a secure, managed deployment

🌿

Elastic Beanstalk

Compute

Managed platform that handles EC2 provisioning, load balancing, auto scaling, and deployments. We use the Immutable deployment policy for zero-risk releases.

🔐

SSM Parameter Store

Secrets Management

Stores all application secrets — DB password, SendGrid API key, Flask secret key — encrypted with KMS. The app reads them at startup via IAM role. No secrets in code.

💻

SSM Session Manager

Access Management

Provides interactive shell access to EC2 instances over HTTPS. Port 22 is permanently closed. No key pairs needed. All sessions logged to CloudWatch automatically.

⚡

SSM Run Command

Operations

Executes shell scripts across all EB instances simultaneously without SSH. Used to restart the app service after secret rotation — one command, all instances.

🗄️

RDS PostgreSQL

Database

Managed relational database storing workshop registrations. Runs inside the same VPC as EB instances. Connection credentials stored in SSM, never in EB config.

📧

SendGrid API

Email Service

Third-party email delivery service. Sends confirmation emails with registration ID and workshop meeting link. API key stored in SSM Parameter Store as SecureString.

Why Not EB Environment Variables?

❌ EB Environment Variables (wrong way)
Visible in EB console to anyone with access
No encryption — DB password in plaintext
Shows up in screenshots, support tickets
✅ SSM Parameter Store (our way)
SecureString = KMS encrypted at rest
Access controlled by IAM role only
CloudTrail logs every single read
Rotate secrets without redeployment
Slide 05 — Zero SSH Infrastructure

SSM Session Manager

Shell access to EC2 instances without opening port 22 — ever

❌ Traditional SSH

Your Laptop ──port 22──► EC2

Requires:
• Port 22 open in security group
• SSH key pair (.pem file)
• Key pair management
• Risk: port 22 = attack surface

✅ SSM Session Manager

Your Laptop ──HTTPS──► AWS SSM ──Agent──► EC2

Requires:
• IAM permission: ssm:StartSession
• SSM Agent (pre-installed on AL2023)
• Port 22: permanently CLOSED
• No key pair needed
🔒

No Open Ports

Port 22 is closed. The instance never accepts inbound connections.

📋

Auto Audit Logs

Every session is automatically logged to CloudWatch. Full audit trail.

🎯

IAM Controlled

Access controlled by IAM policies, not SSH keys. Centralized management.

Slide 06 — Deployment Strategy

Immutable Deployment Policy

The safest deployment strategy — new instances are tested before old ones are removed

❌ All-at-Once

All instances updated simultaneously. If it breaks, all users are affected. No rollback.

⚠️ Rolling

Instances updated in batches. Some users get old version, some get new. Partial outage risk.

✅ Immutable

Brand new instances launched with new code. Health checked. Only then old instances removed. Automatic rollback if health fails.

How Immutable Deployment Works

1. New temp ASG created
→
2. New instances launched
→
3. Health checks run
→
4. ✅ Pass → old instances terminated
→
5. ❌ Fail → temp ASG deleted, old instances keep running
Slide 07 — Application

Workshop Registration App Flow

Micro Degree — Elastic Beanstalk & SSM Workshop registration system

Startup Flow (EC2 boots)

1
EC2 starts → gunicorn runs application.py
2
load_config_from_ssm() reads all 7 SSM parameters using IAM role
3
DB URL assembled in memory: postgresql+pg8000://user:pass@host/db
4
SQLAlchemy connects to RDS PostgreSQL on port 5432
5
registrations table created if not exists
6
/health returns 200 → EB marks instance healthy ✅

Registration Flow (user submits form)

1
User fills Name, Phone, Email on /dashboard
2
Backend generates UUID registration_id — user never types this
3
INSERT into RDS: registration_id, name, phone, email, timestamp
4
SendGrid API called with API key from SSM — sends confirmation email
5
Email contains Registration ID + Meeting Link (from EB env var)
6
/dashboard/records shows all entries with timestamps
Slide 08 — Operations

SSM Run Command

Restart the app across ALL instances simultaneously — no SSH, no manual work

How It Works

# Send restart command to ALL EB instances aws ssm send-command \ --document-name "MyApp-RestartWebService" \ --targets "Key=tag:elasticbeanstalk: environment-name,Values=myapp-prod" \ --parameters "ServiceName=web" # Result per instance: i-0abc123 Success web is ACTIVE i-0xyz456 Success web is ACTIVE

When to Use Run Command

Secret rotation: Updated SendGrid API key in SSM → restart all instances to reload the new key

Meeting link change: Updated WORKSHOP_MEETING_LINK in EB env vars → restart to pick up new value

Service recovery: App crashed on one instance → restart without redeployment

Targets by tag: EB tags every instance with environment name → one command hits all instances automatically
